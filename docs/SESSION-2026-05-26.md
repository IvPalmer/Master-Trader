## Insiders receiver — 4 rounds of codex review

Started from a "ready for session, fixing in parallel" frame; codex's
first pass flagged 6 P0 / 3 P1 / 1 P2 in the execution layer (HOLD).
Worked through 4 review rounds; final verdict GO with high confidence on

production smoke.

### Round 1 — execution layer rewrite (6 P0 / 3 P1 / 1 P2)

- **force_enter payload** was LIMIT-shaped (`stake_amount` / `price` /
  `order_type=limit`). Rewrote as MARKET with flat field names matching
  working Killers (`stakeamount` / `ordertype: market`, no `price`).
- **force_exit partial close** was sending pct as base-coin amount
  (`amount_pct=50` → `"amount": 0.5` BTC). Fixed: query `/trade/{id}`,
  send `amount = current_amount * pct/100`. Added `get_trade()` helper.
- **Graph mutate-on-FT-failure** in `_handle_management`. Wrapped
  close_full/close_partial behind `_ft_response_ok` gate (2xx + no
  `error` key). FT failure now leaves position open.
- **Crash recovery** — pre-create position as `status='requested'`
  BEFORE /forceenter, finalize to 'open' on success. Reconciler back-links
  orphans to FT trades by (pair, side).
- **is_strict_open type-tight** — was checking key presence only; now
  enforces numeric+positive sl, canonical direction set, numeric+positive
  entry or two-number entry_range, direction-valid SL side. Bool exclusion
  explicit (Python `bool` is `int`).
- **Wrong-side SL pre-reject** in `_handle_open` before sizing or order.
- **Increase events** now explicitly rejected on live path (was silently
  logging — replay PnL would diverge from live).
- **Direction-aware target resolution** — `latest_open_position(symbol)`
  ignored direction; long+short on same coin would close wrong side.
- **MNT skip ordering** — skip-list check moved before any Binance call.

### Round 2 — emergent issues from round 1 (3 P0 / 2 P1)

- **Atomic msg_id claim** — `msg_seen` + `record_raw_event` had a window
  where concurrent /event requests for same msg_id could both pass and
  double-fire forceenter. Replaced with `graph.claim_msg()` returning
  rowcount-based bool; SQLite PRIMARY KEY on raw_events.msg_id is the
  atomic gate. `complete_claim()` fills the classification post-classify.
- **Finalize guard** — `_handle_open` finalized on HTTP < 400 even
  without trade_id. Switched to `_ft_response_ok` + isinstance(int, not
  bool) check; `finalize_requested_position` raises ValueError on
  None/non-int.
- **Orphan healing tightened** — was DESC oldest-first wrong direction +
  no ambiguity refusal. ASC oldest-first now; refuses N↔M and N↔1
  ambiguity; 5s age cutoff to avoid racing in-flight finalize.
- **Reply-chain status filter** — `_position_by_id` returned positions of
  any status; closed/failed/requested openers could be hit by reply.
  Added `JOIN positions WHERE status='open'`.
- **Cold-path validator** — new `validate_open_payload()` runs at top of
  `_handle_open` BEFORE any field access. Type-checks
  symbol/direction/sl/entry/entry_range/tp.

### Round 3 — accidental exposure-orphaning (2 P0)

Round 2's finalize-guard introduced a subtle bug: marking position
`failed` on exception or 2xx-no-trade-id meant FT-side exposure could be
orphaned past reconciler reach.

- **3-state FT classifier** — new `_ft_response_kind(resp)` returns
  `accepted` / `rejected` / `uncertain`. Exception, 5xx, 2xx-no-trade-id
  now all keep position as `requested` for reconciler healing. Only
  definite 4xx or 2xx-with-error marks `failed`.
- **1-orphan ↔ many-FT-trades** — was auto-linking to oldest; refuses
  now (manual/stale exposure could hijack the link).

### Round 4 — operator visibility/control (2 P0)

The "reconciler heals forever" stance was acceptable only if stale
`requested` rows were operator-visible and resolvable.

- `/system` exposes `requested_orphans_count` + `oldest_requested_age_sec`
- `GET /positions/requested` lists stale requested with age, sizing,
  signal context, opener msg_id
- `POST /position/{id}/fail` — manual mark-failed for stale requested
- `POST /position/{id}/link?ft_trade_id=N` — manual link for
  ambiguous-but-confirmed cases
- `finalize_requested_position` enforces ft_trade_id clash detection at
  graph layer (raises ValueError); reconciler catches the race gracefully
