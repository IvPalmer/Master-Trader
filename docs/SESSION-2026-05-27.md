## Round 1 (d2f9cbf) — full hardening sweep

User flagged that close_partial was collapsing into close_full and re-firing
on observer redelivery. Initial codex review found that AND 12 more issues
across 6 rounds. All applied in one commit:

- **Partial-close**: real pct-of-original semantics matching the Insiders
  simulator. `pct_open` column tracks remainder. UNIQUE(pos_id, msg_id, kind)
  constraint + claim-before-await prevents double-fire on redelivery.
  `FULL_CLOSE_RESIDUAL_PCT=0.5` flips position to closed when remainder
  is sub-noise.
- **Target guard**: parse the channel's TARGETS line at open. If all
  targets already crossed by Binance mark → 🚫 skip with reason
  `all_targets_crossed`. Otherwise persist remaining in
  `targets_remaining` JSON column.
- **Slippage gate** (`KILLERS_MAX_ENTRY_SLIPPAGE_PCT=3.0` default):
  LONG skip if mark > entry_hi * 1.03, SHORT skip if mark < entry_lo *
  0.97. HYPE #2144 would have been skipped (mark $60.86 vs entry_hi
  $57 = +6.78% > 3%). Fail-closed when bounds unparseable.
- **Rule fast-path** (`killers_bot/strict_open.py`): regex parser for
  clean OPEN signals. Observer tries rule first → if match, skip the
  7.6s Claude CLI hop → POST receiver immediately. Backtest against
  282 historical messages: 100% precision, 100% recall (after fix for
  single-char tickers like `$T/USDT`).
- **Persistent aiohttp sessions** in receiver lifespan + true
  background notify task.
- **Operator endpoints**: GET /target_orders, GET /system,
  POST /events/pending/{id}/resolve.

Latency projection: signal → fill drops from ~9.3s to ~1.5s on clean opens.

## Round 3 (f15f7ef) — Phase 2 active TP placement

Operator pushed back on codex's "channel as source of truth" concern:
"os hits são só anúncios" — the Killers signaler explicitly publishes
the full TP ladder at open and NEVER manually closes at non-TP prices.
All close_partial msgs in the captured corpus are post-hoc "Target N
hit" announcements matching specific TPs from the open signal.

This invalidated the override-engineering codex was worried about and
unlocked a cleaner Phase 2 design:

- New `target_orders` table with state machine (pending → active →
  filled / cancelled / rejected / skipped)
- At open: divide ft_amount into N=len(remaining_targets) slices,
  POST /forceexit ordertype=limit at each TP price, capture
  ft_order_id from the FT trade snapshot
- Background reconciler ticks every 20s: groups active rows by
  ft_trade_id (ONE /trade/{id} fetch serves all N targets on the same
  trade), transitions states based on order.status
- **close_partial is now audit-only when Phase 2 has live target_orders**
  for the position: inline reconcile FIRST (catches just-filled
  limits), then if any row exists in (pending, active, filled) →
  event response `{status:"audit_only", reason:"active_tp_limits"}`,
  no ft_force_exit call. Avoids the double-exit race.
- close_full unchanged — explicit "close all" / SL hit still goes
  through market exit; pending target_orders rows marked 'cancelled'
  locally (FT closes position → exchange cancels open limits).

Codex GO after 2 review rounds.

## Round 4 (2b51f1a) — production 500 caught by next real signal

At 2026-05-27 19:38:14 UTC, msg 3471 arrived (XLM #2142 close_partial
with 3 TPs hit + 69% profit). Receiver returned 500
`NameError: name 'asyncio' is not defined`.

Root cause: `import asyncio` was only INSIDE `lifespan` (function
scope binding) and the reconcile loops. `handle_event` referenced
`asyncio.create_task(...)` at module scope → NameError on every
/event POST. Unit tests for `_process_event` passed because they
imported asyncio at the test file level; they never went through the
FastAPI handler.

Observer logged the 500 and moved on. No retry, no Telegram alert,
msg lost.

Fix: top-level `import asyncio` in `services/killers-receiver/app/main.py`
+ new `tests/test_handle_event_endpoint.py` (5 tests using
`fastapi.testclient.TestClient(app)` context manager so lifespan fires).
Manual replay of msg 3471 against fixed receiver → 200
`{action:"skipped", reason:"no_active_position (no_match)"}` (we never
opened XLM #2142 — it was opened by Killers before our bot started
observing).

## Round 5 (dfa5a2a) — codex post-mortem follow-ups

Codex flagged 4 issues + 1 enhancement in the post-mortem (thread
019e6aff). All applied in one commit:

**(a) aiohttp module-level cleanup** — kill the `try/except ImportError`
optional pattern + 9 local imports across killers + insiders. aiohttp
is a production dep, fail fast at startup beats latent handler crashes.
Also dropped `_ensure_auth` lazy-build in insiders FreqtradeClient.

**(b) Observer retry on 5xx** — `_post_to_receiver` was best-effort
one-shot. New policy: 3 attempts (0s, 2s±20%, 5s±20% jitter), retry
on exception/5xx/429, never 4xx. Final exhaustion logs at ERROR with
msg_id.

**(c) Ingress audit table** — new `ingress_events` table inserted at
TOP of handle_event BEFORE any processing. Even on handler crash, a
forensic row exists. Catch-all try/except wraps `_process_event` and
returns structured HTTPException(500) with msg_id + kind instead of
opaque internal server error. Added GET /ingress endpoint.

**(d) Sweep insiders** — insiders already had top-level asyncio
(checked). Added 6 TestClient endpoint tests to catch the same class
of handler-only-dep bug going forward.

**(e) Notify-fired test + ingress audit tests** — codex's specific
concern: earlier endpoint tests proved `asyncio.create_task` didn't
crash, didn't prove notify actually fired. New test patches
`_notify_telegram` with a spy + asserts call count ≥ 1 after the
event triggers a formatted alert. 3 more tests cover ingress audit
row on success + on crash + /ingress endpoint listing.
