## 2. WEEX integration — BUILT + live-validated ($0.27 total burn)

Bypassed ccxt entirely (WEEX unsupported, issue #27680 dead). Wrote a custom
REST client. All live-tested on the operator's real WEEX account.

- **`weex_probe/weex_client.py`** — OKX-style HMAC sign (key+secret+passphrase),
  19 endpoints. Auth worked first try; Mac IP allowlisted.
- **Atomic bracket**: `POST /capi/v3/order` MARKET + attached `slTriggerPrice`
  + `tpTriggerPrice` → WEEX auto-spawns STOP_MARKET + TAKE_PROFIT_MARKET algos,
  both `reduceOnly`. No naked-position window (resolved codex's main concern).
- **One-way (COMBINED) mode is MANDATORY.** Hedge (SEPARATED) silently breaks
  every management endpoint (`-1054 "Position ID missing"`, all field-name
  variants tried). In one-way: partial close via `/order` opposing side,
  SL move via `/modifyTpSlOrder`, full close via `/closePositions`.
- **Idempotency**: WEEX dedups by `newClientOrderId` server-side (same COID →
  same orderId, verified no double-fill). **Rate limit**: 15 concurrent
  /openAlgoOrders in 971ms, no 429.
- **Partial-close → SL/TP algos don't auto-resize**, but `reduceOnly` caps the
  trigger at remaining size → still safe.
- **KYC + Google Authenticator 2FA + fund code (`314159`)** set up.
- **Custody test PASSED**: $5 WEEX → Binance via Polygon, <3 min, $0.20 fee,
  credited "Concluído". Txid `0xc6cc321fb02abe08cbe5fc0192b2d441c8542277e53101bff3b7647d5e7fed60`.

Verdict: the plumbing genuinely works. The blocker is economics + venue trust,

## 3. The backtest validation — the session's core finding

### Three numbers existed for the same 83-day Eduardo export (428 msgs, 147 trades)

| Number | Source | Status |
|---|---|---|
| +76.4% | `replay/trades_llm_2026-05-26.json` | **simulator bug** |
| +24.8% | agent re-walk `weex_probe/replay_on_weex.py` | initial-SL/TP-only, no scale-out modeling |
| **+9.77%** | **`weex_probe/proper_backtest.py`** (event-driven, clean Binance 1m) | **authoritative** |

### Why +76% was fake

The original sim (`ft_userdata/insiders_bridge/_local/weex.py::resolve_exits`):
1. **Ignored all partial closes + SL moves** — collapsed each managed trade to
   one entry→exit line. Dennis's entire edge is the scale-out + trailing stop;
   the sim modeled none of it.
2. **Phantom mark-to-end exits** — positions never explicitly closed got
   stamped the LAST dataset price (~76,089) regardless of date. The 3 biggest
   "wins" in the dataset ($77 + $69 + $62 = $209) were all this phantom.

### Smoking gun — msg 79 (BTC LONG)

- Sim claimed: entry 66,500 → exit **76,095** → **+$77.58** (biggest "win")
- Real Binance OHLCV (Feb 11-13 2026): BTC high **69,958** (never near 76k);
  SL 65,300 **was hit** (low 65,081)
- Dennis's own words in the channel: "full close at **65,200**"
- Proper sim: **+$10.86** (scaled out at the ~69,958 pop, trail caught the drop)

### The honest number

`proper_backtest.py` processes the full event stream per trade (entry →
walk OHLCV for SL/TP between events → partial closes at event-time price →
trailing SL moves → increases → close_full → blended PnL). On clean Binance
1m data:

- **+9.77% / 83 days**, WR 45.5%, PF 1.22
- Stable under partial-default assumption: +9.56% / +9.77% / +10.21% (25/33/50%)
- 46 of 99 trades exit on the trailing stop (the edge is thin)
- This is the **faithful-mirror CEILING**; live latency/slippage/fees erode it
  to ~+6-10%/6mo

WEEX prices ≈ Binance (agent confirmed within **$0.64 / +0.07pp** over 83d on
cached WEEX klines — no "WEEX tax"). NOTE: re-running `proper_backtest.py weex`
returned garbage ($0/58 no_data) because WEEX's live klines endpoint won't
serve 3-month-old 1m history via startTime pagination. Use the cached-kline
comparison for WEEX, not a fresh fetch.
